## 1. 核心设计原则

### 1.1 统一标准化（第一性原则）

系统的每一层、每一个组件必须遵循**同一套接口规范**。没有例外。

| 维度 | 标准化要求 | 反例（不这么做会怎样） |
|------|-----------|----------------------|
| 组件接口 | 所有组件实现相同的 `IComponent` 接口 | 每个组件自己定义通信协议 → 系统耦合 |
| 数据格式 | 所有跨组件数据使用标准 swbn 格式 | 组件间需要不断做格式转换 → 性能损耗 |
| 生命周期 | 所有组件共享相同的启动/暂停/恢复/停止状态机 | 组件管理混乱 → 资源泄漏 |
| 错误处理 | 所有错误使用统一 `SsiError` 类型 | 错误码满天飞 → 调试噩梦 |
| 通信方式 | 所有组件间通信经过标准 IPC 总线 | 点对点连接 → 蜘蛛网状拓扑 |

### 1.2 浏览器即内核（第二性原则）

**浏览器不是用户空间应用——它是系统内核的渲染与脚本引擎。**

在 AI-TP OS 中：
- **Chromium 渲染引擎** → 系统窗口管理器 + UI 框架
- **V8 JavaScript 引擎** → 系统脚本层 + 策略引擎
- **WASM 运行时** → 原生应用执行引擎
- **WebGPU** → 系统图形 API
- **WebSocket / WebRTC** → 系统网络 API
- **Web Worker** → 系统线程调度器
- **OPFS (Origin Private File System)** → 用户空间文件系统
- **Service Worker** → 系统后台服务管理器

> 这不是"在一个操作系统里运行一个浏览器"——  
> 这是**以浏览器架构为基座的操作系统**。

### 1.3 全平台同构（第三性原则）

同一套标准化架构，在所有平台上以相同的方式运行：

```
┌─────────────────────────────────────────────────────────┐
│                   AI-TP 应用                             │
│              (标准 WASM 组件，.swbn 格式)                  │
├─────────────────────────────────────────────────────────┤
│                AI-TP OS 标准化层                         │
│  ┌───────────────┐  ┌───────────────┐  ┌────────────┐  │
│  │ 标准接口 (SSI) │  │ 标准组件模型   │  │ 标准 IPC   │  │
│  └───────┬───────┘  └───────┬───────┘  └──────┬─────┘  │
├──────────┼──────────────────┼──────────────────┼────────┤
│  ┌───────┴──────────────────┴──────────────────┴─────┐  │
│  │         平台适配层 (Adapter Layer)                  │  │
│  ├────────────────┬────────────────┬──────────────────┤  │
│  │  Browser Mode  │  Native Mode   │  Embedded Mode   │  │
│  │  (Chrome/V8)   │  (Linux/GLIBC) │  (RTOS/MCU)     │  │
│  └────────────────┴────────────────┴──────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

| 平台模式 | 运行时 | 目标设备 |
|----------|--------|---------|
| **Browser Mode** | Chromium V8 + WASM | 所有浏览器、PWA、WebView |
| **Native Mode** | Linux/glibc + WASM | 服务器、桌面、Android |
| **Embedded Mode** | 微型 WASM 运行时 | IoT、路由器、MCU |

**同一份应用组件无需修改，在所有模式下一致运行。**

## 3. 标准化层级定义

### 3.1 层级模型

每一层有严格定义的接口边界。跨层通信只能通过标准接口进行。

| 层级 | 编号 | 职责 | 对外接口 | 对内依赖 |
|------|------|------|---------|---------|
| 应用层 | L5 | 用户态应用运行 | SSI-API | 系统服务层 |
| 系统服务层 | L4 | 提供标准系统服务 | SSI-SVC | 内核层 |
| 内核层 | L3 | 运行时、安全、IPC | SSI-KRN | 硬件抽象层 |
| 硬件抽象层 | L2 | 平台适配 | SSI-HAL | 物理硬件 |
| 物理层 | L1 | 实际硬件 | — | — |

### 3.2 层间通信规则

```
┌──────┐
│  L5  │ ←→ 只能通过 SSI-API 与 L4 通信
├──────┤
│  L4  │ ←→ 通过 SSI-SVC 向 L5 提供服务，通过 SSI-KRN 与 L3 通信
├──────┤
│  L3  │ ←→ 通过 SSI-HAL 与 L2 通信
├──────┤
│  L2  │ ←→ 直接操作物理硬件
└──────┘
```

**禁止规则**：
- L5 不能直接调用 L3/L2/L1 接口
- L4 不能绕开 L3 直接操作硬件
- 所有跨层调用必须经过 SSI 接口层

## 4. 统一接口规范 (SSI)

### 4.1 SSI 核心规范

**System Standard Interface (SSI)** 是系统的"宪法"。所有组件必须实现。

```c
// ─── SSI 组件基础接口（所有组件必须实现）───

// 组件标识
typedef struct {
    uint8_t  uuid[16];           // 全局唯一 ID
    char     name[64];           // 组件名
    char     version[16];        // 语义化版本
    uint32_t api_version;        // SSI API 版本号 (当前: 1)
    char     type[32];           // 组件类型: "browser", "storage", "compute"...
} SsiComponentId;

// 组件状态机
typedef enum {
    SSI_STATE_UNINITIALIZED = 0,
    SSI_STATE_INITIALIZING,
    SSI_STATE_READY,
    SSI_STATE_RUNNING,
    SSI_STATE_SUSPENDED,
    SSI_STATE_ERROR,
    SSI_STATE_TERMINATED
} SsiComponentState;

// 组件生命周期接口（必须实现）
typedef struct SsiComponentVtbl {
    // ── 生命周期 ──
    int  (*init)(void* ctx, const SsiConfig* config);
    int  (*start)(void* ctx);
    int  (*suspend)(void* ctx);
    int  (*resume)(void* ctx);
    int  (*stop)(void* ctx);
    void (*destroy)(void* ctx);

    // ── 查询 ──
    SsiComponentState (*get_state)(void* ctx);
    SsiComponentId    (*get_id)(void* ctx);
    int               (*get_status)(void* ctx, SsiStatus* out);

    // ── 通信 ──
    int (*send_message)(void* ctx, const SsiMessage* msg);
    int (*receive_message)(void* ctx, SsiMessage* out, uint32_t timeout_ms);
} SsiComponentVtbl;
```

### 4.2 标准数据类型

所有跨组件数据使用统一的 swbn 格式（Standardized WASM Binary Notation）：

```
swbn 数据包结构:
┌──────────────────────────────────────────────────┐
│ Magic: "SWBN" (4 bytes)                          │
│ Version: u32 (4 bytes)                           │
│ Type: u32 (4 bytes)                              │
│   - 0x01: 系统消息                                │
│   - 0x02: 组件数据                                │
│   - 0x03: 文件数据                                │
│   - 0x04: 流数据                                  │
│   - 0x05: 事件                                    │
│ Payload Length: u64 (8 bytes)                    │
│ Payload: [...]                                   │
│ Checksum: SHA256 (32 bytes)                      │
└──────────────────────────────────────────────────┘
```

### 4.3 标准错误处理

```c
// 统一错误类型
typedef struct {
    uint32_t    code;        // SSI 错误码
    char        message[256];// 人类可读消息
    char        component[64];// 源组件
    uint64_t    timestamp;   // 时间戳
    uint8_t     severity;    // 0=debug, 1=info, 2=warn, 3=error, 4=fatal
} SsiError;

// 预定义错误码
#define SSI_OK               0x00000000
#define SSI_ERR_GENERIC      0x80000001  // 通用错误
#define SSI_ERR_NOT_FOUND    0x80000002  // 组件未找到
#define SSI_ERR_BUSY         0x80000003  // 组件忙
#define SSI_ERR_TIMEOUT      0x80000004  // 超时
#define SSI_ERR_PERMISSION   0x80000005  // 权限不足
#define SSI_ERR_INVALID      0x80000006  // 无效参数
#define SSI_ERR_NOT_SUPPORTED 0x80000007 // 不支持的操作
#define SSI_ERR_MEMORY       0x80000008  // 内存不足
```

## 5. 组件模型

### 5.1 标准化组件定义

每个系统组件是一个 **WASM 模块**，遵循 SSI 接口规范。组件打包为 `.swbn` 格式：

```
┌───────────────────────────────┐
│  .swbn 包结构 (Standard WASM  │
│   Binary Notation Archive)    │
├───────────────────────────────┤
│  📋 manifest.json              │
│     - id, name, version        │
│     - entry: "main.wasm"       │
│     - permissions: [...]       │
│     - dependencies: [...]      │
│     - interfaces: [...]        │
├───────────────────────────────┤
│  🧩 main.wasm                  │
│     - 实现 SSI 接口的 WASM 模块 │
│     - 导出的标准函数            │
├───────────────────────────────┤
│  📁 assets/                    │
│     - 资源文件                  │
├───────────────────────────────┤
│  🖼️ icon.png                   │
├───────────────────────────────┤
│  🔏 signature.sig              │
│     - 数字签名                  │
└───────────────────────────────┘
```

### 5.2 系统内置组件

| 组件名 | 类型 | 接口 | 说明 |
|--------|------|------|------|
| `browser-engine` | `browser` | SSI-BR | Chromium 渲染引擎，系统原生 UI 层 |
| `window-manager` | `ui` | SSI-UI | 窗口系统，基于 WebGPU 的合成器 |
| `storage-engine` | `storage` | SSI-DB | P2P 存储引擎 (libai-storage) |
| `compute-engine` | `compute` | SSI-AI | AI 计算调度引擎 |
| `network-stack` | `network` | SSI-NET | AI-TP 协议栈 |
| `security-module` | `security` | SSI-SEC | 安全与身份管理 |
| `wasm-runtime` | `runtime` | SSI-KRN | WASM 执行引擎 |
| `file-system` | `fs` | SSI-FS | 虚拟文件系统 |
| `device-manager` | `hal` | SSI-HAL | 硬件抽象层 |

### 5.3 组件生命周期

```
                      ┌─────────┐
                      │ 未初始化  │
                      └────┬────┘
                           │ init()
                           ▼
                      ┌─────────┐
                      │ 初始化中  │ ←──── error → ┌───────┐
                      └────┬────┘                │ 错误   │
                           │ init() 完成          └───────┘
                           ▼                         │
                      ┌─────────┐                   │
                      │  就绪    │ ←── retry ────────┘
                      └────┬────┘
                           │ start()
                           ▼
 ┌────────┐          ┌─────────┐
 │ 暂停    │ ←─────── │ 运行中   │
 └────┬───┘  suspend  └────┬────┘
      │                    │ stop()
      │ resume()           ▼
      │              ┌─────────┐
      └─────────────→│ 已终止   │
                     └─────────┘
```

### 6.2 浏览器集成接口

所有浏览器功能通过标准 SSI-BR 接口暴露给系统其他组件：

```c
// SSI-BR: 浏览器引擎标准接口

// ── 渲染（窗口系统基座）──
int ssi_browser_render(
    const char* url,
    SsiRect* target,       // 渲染目标区域
    SsiRenderOptions* opts // 渲染选项（GPU 加速、透明度等）
);

// ── 脚本执行（系统策略引擎）──
int ssi_browser_execute(
    const char* script,     // JavaScript/WAT 代码
    SsiScriptContext* ctx,  // 执行上下文
    SsiValue* result        // 返回值
);

// ── 计算（GPU 加速）──
int ssi_browser_compute(
    const char* shader,     // WebGPU/WGSL shader
    SsiBuffer* input,
    SsiBuffer* output
);

// ── 网络（AI-TP 协议栈前端）──
int ssi_browser_network(
    SsiNetworkRequest* request,   // 统一网络请求格式
    SsiNetworkResponse* response
);

// ── 存储（虚拟文件系统前端）──
int ssi_browser_storage(
    SsiStorageOp op,         // read/write/delete/list
    const char* key,
    SsiBuffer* value
);

// ── 安全（沙箱管理）──
int ssi_browser_sandbox(
    SsiSandboxConfig* config,
    SsiSandboxHandle* handle
);
```

### 6.3 三模渲染架构

浏览器引擎在系统中以三种模式运行：

| 模式 | 用途 | 技术栈 | 资源消耗 |
|------|------|--------|---------|
| **全模式** | 完整桌面/移动 UI | Chromium + WebGPU + WASM | 高 |
| **轻量模式** | 嵌入式/信息显示 | Skia 软件渲染 + 简化 DOM | 中 |
| **无头模式** | 服务器/后台任务 | V8 + WASM 运行时 | 低 |

系统根据设备性能自动选择渲染模式，应用代码无需修改。

## 7. 标准化数据流

### 7.1 系统数据流全景

```
用户输入
    │
    ▼
┌─────────────────────────────────────────────────────┐
│ 浏览器引擎 (SSI-BR)                                 │
│  1. 接收输入事件                                     │
│  2. 通过 SSI 总线转发到目标组件                        │
│  3. 组件处理完成后返回结果                             │
│  4. 浏览器引擎渲染结果                                │
└─────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────┐
│             SSI 标准服务总线                           │
│                                                     │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐          │
│  │ 网络请求  │  │ 存储请求  │  │ 计算请求  │          │
│  │ ↓        │  │ ↓        │  │ ↓        │          │
│  │ SSI-NET  │  │ SSI-DB   │  │ SSI-AI   │          │
│  │ → AI-TP  │  │ → P2P FS │  │ → 调度器 │          │
│  └──────────┘  └──────────┘  └──────────┘          │
└─────────────────────────────────────────────────────┘
```

### 7.2 标准化请求/响应流程

```
应用组件                    SSI 总线                  目标组件
   │                         │                         │
   │  1. 封装 SSI 请求        │                         │
   │────────────────────────→┤                         │
   │                         │  2. 路由到目标组件        │
   │                         │────────────────────────→│
   │                         │                         │  3. 处理请求
   │                         │                         │──────
   │                         │                         │  ← 完成
   │                         │  4. 返回标准 SSI 响应     │
   │                         │←────────────────────────│
   │  5. 接收标准响应         │                         │
   │←────────────────────────┤                         │
   │                         │                         │
```

### 7.3 数据格式统一

所有数据交换使用三种统一格式：

| 格式 | 用途 | 序列化 |
|------|------|--------|
| **swbn** | 结构化数据交换 | 二进制（Schema 驱动） |
| **swbn-stream** | 流式数据（实时音视频/日志） | 分片二进制 |
| **swbn-event** | 事件通知 | 轻量二进制事件包 |

## 8. 安全模型

### 8.1 分层安全

```
┌──────────────────────────────────────────────────┐
│  L5: 应用层沙箱                                    │
│  • 每个 .swbn 组件运行在独立 WASM 沙箱              │
│  • 无直接系统调用权限                               │
│  • 通过 SSI 接口进行所有操作                        │
├──────────────────────────────────────────────────┤
│  L4: 服务层权限控制                                 │
│  • 服务间的访问控制列表 (ACL)                       │
│  • 每个 SSI 调用经过权限检查                         │
├──────────────────────────────────────────────────┤
│  L3: 内核层安全                                     │
│  • WASM 运行时隔离                                  │
│  • 内存保护                                         │
│  • IPC 审计日志                                     │
├──────────────────────────────────────────────────┤
│  L2: 硬件层安全                                     │
│  • TPM/安全芯片集成                                  │
│  • 安全启动链                                       │
└──────────────────────────────────────────────────┘
```

### 8.2 组件权限模型

每个 `.swbn` 组件在其 manifest 中声明所需权限：

```json
{
  "component": "my-ai-app",
  "version": "1.0.0",
  "permissions": [
    {
      "interface": "SSI-NET",
      "operations": ["fetch", "websocket"],
      "restriction": "outbound-only"
    },
    {
      "interface": "SSI-DB",
      "operations": ["read", "write"],
      "paths": ["/user-data/*"]
    }
  ]
}
```
