## 🧠 设计哲学

### 核心原则

1. **所有设备通用**  
   - 手机、电脑、IoT、服务器都能运行
   - 资源受限设备（如路由器）也能作为轻节点

2. **互联网有的，它都要有**  
   - TCP/IP → AI-TP 协议栈
   - DNS → 去中心化寻址
   - HTTP → P2P 内容分发
   - 文件系统 → 去中心化存储

3. **互联网没有的，它也要有**  
   - AI 原生：协议层内置 AI 任务分发
   - 去中心化：无单点故障
   - 自组织：节点自动发现、自动组网

4. **安全优先**  
   - 端到端加密（默认开启）
   - 零信任架构
   - 存储证明（防止作弊）

### 三层职责

| 层级 | 名称 | 核心职责 | 类比传统系统 |
|------|------|----------|--------------|
| **网络层** | Network Manager | 节点发现、NAT 穿透、去中心化寻址、P2P 通信 | TCP/IP 协议栈 + DNS |

| **计算层** | Compute Manager | AI 任务调度、模型分发、隐私计算、激励机制 | 操作系统调度器 + GPU 驱动 |

## 🏗️ 系统标准架构 (SSI)

> AI-TP OS 不是一堆工具的拼凑——它是一个**完全标准化的底层系统**。  
> 所有组件遵循同一套接口规范。浏览器不是移植目标，而是系统的原生核心。  
> **详细规范参见**: [SYSTEM-STANDARD.md](SYSTEM-STANDARD.md) | [SPEC-INTERFACE.md](SPEC-INTERFACE.md)

### 标准化层级

```
┌─────────────────────────────────────────────────────────┐
│   L5: 应用层 (Application)                                │
│   WASM 应用 / 系统工具 / 第三方组件 (.swbn 格式)          │
│   接口: SSI-API                                          │
├─────────────────────────────────────────────────────────┤
│   L4: 系统服务层 (System Services)                        │
│   浏览器引擎 / 窗口系统 / 存储引擎 / 计算引擎 / 网络协议栈 │
│   接口: SSI-SVC (SSI-BR, SSI-UI, SSI-DB, SSI-AI, SSI-NET)│
├─────────────────────────────────────────────────────────┤
│   L3: 内核层 (Kernel)                                    │
│   WASM 运行时 / SSI 总线 / 组件管理器 / 安全模块          │
│   接口: SSI-KRN, SSI-SEC                                 │
├─────────────────────────────────────────────────────────┤
│   L2: 硬件抽象层 (HAL)                                   │
│   设备信息 / 传感器 / GPU / 网络状态 / 电池               │
│   接口: SSI-HAL                                          │
├─────────────────────────────────────────────────────────┤
│   L1: 物理层 (Physical)                                  │
│   手机 / 电脑 / 浏览器沙箱 / 服务器 / IoT / 路由器       │
└─────────────────────────────────────────────────────────┘
```

### 设计原则

| # | 原则 | 含义 |
|---|------|------|
| 1 | **统一标准化** | 每一层、每一个组件遵循同一套 SSI 接口规范，无例外 |
| 2 | **浏览器即内核** | 浏览器引擎是系统的原生渲染与脚本执行引擎，不是用户空间应用 |
| 3 | **全平台同构** | 同一套标准化架构在 Browser / Native / Embedded 模式一致运行 |
| 4 | **组件即 WASM** | 所有应用和系统服务都是 `.swbn` 格式的 WASM 组件 |
| 5 | **接口即契约** | SSI 接口是硬边界——L5 不能直接调用 L3 |

### 组件模型

```
my-component-1.0.0.swbn
├── manifest.json          # 组件清单（声明接口、权限、依赖）
├── main.wasm              # WASM 模块（实现 SSI 接口）
├── main.js                # Emscripten JS 胶水（可选）
├── types/                 # 类型定义
├── assets/                # 资源文件
└── signature.sig          # 数字签名
```

所有组件通过 **SSI 标准服务总线**通信——统一的消息格式、路由、安全策略。

```
 AI-TP 系统层

│  ┌─────────────────────┐  ┌──────────────────────┐ │

│  └─────────────────────┘  └──────────────────────┘ │

```

### 三模渲染

| 模式 | 技术栈 | 场景 |
|------|--------|------|
| **全模式** | Chromium + WebGPU | 桌面/移动端全功能 UI |
| **轻量模式** | Skia + Canvas 2D | 嵌入式/ IoT 信息显示 |

## 🌐 网络层设计

### 功能模块

#### 1. 节点发现 (Node Discovery)

**目标**：让节点找到彼此（替代 DNS + DHCP）

**方案**：混合发现
```
1. 先尝试 mDNS（局域网）
2. 如果失败，查询本地缓存（AI 预测的高概率节点）
3. 如果还失败，使用 DHT 广播
```

#### 2. NAT 穿透 (NAT Traversal)

**目标**：让内网节点可以直接通信（替代 STUN/TURN/ICE）

**AI-TP 优化方案**：
```
1. 预测 NAT 类型（基于历史数据）
2. 如果是全锥型 NAT → 直接用 STUN
3. 如果是对称 NAT → 用 AI 预测的端口猜测
4. 如果打洞失败 → 用 P2P 中继（去中心化中继池）
```

**关键创新**：
- **去中心化中继池**：任何节点都可以成为中继，获得积分奖励
- **AI 端口预测**：基于历史打洞数据，预测对称 NAT 的端口变化规律

#### 3. 去中心化寻址 (Decentralized Addressing)

**目标**：替代 IPv4/IPv6（中心化分配）

**方案**：
```
设备 ID = 公钥的哈希（类似 IPFS 的 PeerID）
- 格式：ai-tp:<base58-encoded-public-key>
- 示例：ai-tp:QmYwAPJzv5CZsnA7RdNZ7Z8G6g6Z5J7J7X5X5X5X5X5X5X5

优点：
1. 设备自主生成 ID（不需要中心机构分配）
2. ID 就是公钥（内置身份认证）
3. 支持移动性（设备换网络，ID 不变）
```

#### 4. 传输层优化 (Transport Layer)

**目标**：替代 TCP（可靠但慢）/ UDP（快但不可靠）

**AI-TP 传输层**：
```
AI-TP Transport Protocol (ATP)：
1. 基于 QUIC（0-RTT 握手）
2. 内置 P2P 优化（多路径支持）
3. 自适应重传（AI 预测丢包）
4. 内置加密（不需要 TLS）
```

## 💾 存储层设计

### 功能模块

#### 1. 统一存储 API (Unified Storage API)

**目标**：让应用不需要关心数据存在本地还是 P2P 网络

**API 设计**（类似 POSIX 文件系统）：
```c
// 存储对象
int ai_storage_put(const char *key, const void *data, size_t len);

// 读取对象
int ai_storage_get(const char *key, void *data, size_t *len);

// 删除对象
int ai_storage_delete(const char *key);

// 列出对象（支持前缀匹配）
int ai_storage_list(const char *prefix, char **keys, size_t *count);
```

**存储后端**：
- **本地存储**：本地磁盘（快速、私密）
- **P2P 存储**：分布式哈希表（DHT）存储
- **混合存储**：热数据本地，冷数据 P2P

#### 2. P2P 文件同步 (P2P File Sync)

**目标**：让文件在多个设备间自动同步（类似 Syncthing）

**方案**：
```
基于内容寻址（类似 IPFS）：
1. 文件分块（每块 256KB）
2. 计算每块的哈希（SHA256）
3. 将块分布到多个节点（冗余存储）
4. 节点故障时，自动修复（从其他节点拉取副本）
```

#### 3. 存储证明 (Storage Proof)

**目标**：防止节点作弊（声称存储了数据，但实际没有）

**方案**（轻量级）：
```
1. 挑战者发送随机数
2. 存储节点用随机数和数据计算哈希
3. 如果哈希正确，证明数据还在
4. 不需要区块链（用本地账本即可）
```

## 🧮 计算层设计

### 功能模块

#### 1. AI 任务调度器 (AI Task Scheduler)

**目标**：将 AI 任务分配到空闲节点

**调度策略**：
```
输入：
- 任务需求（CPU/GPU/内存）
- 节点资源（可用 CPU/GPU/内存）
- 网络延迟（任务数据需要传输）

输出：
- 最优节点列表

策略：
1. 先过滤：资源不足的节点
2. 再排序：延迟低 → 优先级高
3. 最后分配：任务拆分（大任务拆成多个小任务）
```

#### 2. 模型分发 (Model Distribution)

**目标**：将大模型分片存储在不同节点

**方案**：
```
模型分片：
1. 将模型权重分片（每片 100MB）
2. 将分片分布到多个节点
3. 推理时，并行从多个节点拉取分片
4. 聚合结果（类似分布式推理）
```

#### 3. 隐私计算 (Privacy-Preserving Computation)

**目标**：数据不出本地，只传梯度（联邦学习）

**方案**：
```
联邦推理：
1. 本地数据不离开设备
2. 只上传模型梯度（加密）
3. 中心节点聚合梯度，更新模型
4. 下发更新后的模型
```

#### 4. 激励机制 (Incentive Mechanism)

**目标**：让节点愿意贡献资源

**积分系统**：
```
赚取积分：
- 贡献存储空间 → 获得存储积分
- 贡献计算资源 → 获得计算积分
- 作为中继节点 → 获得网络积分

消耗积分：
- 存储数据 → 支付存储积分
- 运行 AI 任务 → 支付计算积分
- 使用 P2P 网络 → 支付网络积分
```

## 🔗 三层交互

### 典型工作流程

#### 场景 1：AI 推理任务

```
用户发起推理请求
    ↓
计算层：调度器选择最优节点
    ↓
网络层：将推理数据传到目标节点
    ↓
计算层：目标节点执行推理
    ↓
存储层：缓存推理结果（加速下次查询）
    ↓
网络层：将结果返回给用户
```

#### 场景 2：文件存储

```
应用调用存储 API
    ↓
存储层：判断文件大小和重要性
    ↓
存储层：选择存储后端（本地/P2P/云）
    ↓
网络层：如果是 P2P 存储，将文件分片传输到多个节点
    ↓
存储层：记录元数据（文件→分片映射）
    ↓
存储证明：定期验证分片还在
```
